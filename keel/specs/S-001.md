# Sentinel Atlas Compatibility + Migration

## Objective

Keep Sentinel as the validation and audit layer while Atlas becomes the
surface catalog, capture, and review producer.

## Product Model

Sentinel should own:

- validation of expected coverage
- visual diff and parity checks
- doctor and CI-facing audit behavior
- compatibility and migration from older proof contracts

Sentinel should not own:

- the primary surface catalog authoring workflow
- the review dashboard as a product in its own right
- product-specific routing logic that belongs in Atlas or the app

## In Scope

- normalize config vocabulary and platform naming
- define Atlas adapter types for surfaces, scenarios, paths, and targets
- add migration-oriented command contracts and fixtures
- support session-based capture boundaries where Atlas hands off artifacts
- upgrade dashboard grouping beyond flat screen slugs
- extend validation and parity paths to Atlas artifacts
- document the transition path for products already using `catalog.screens`

## Out Of Scope

- replacing Atlas as the review product
- deleting the old catalog path before products migrate
- reintroducing design token generation responsibilities

## Definition Of Done

A Sentinel Atlas-migration task is done only when:

- the command and type contracts are explicit
- backward compatibility is stated and tested
- products can understand whether Sentinel expects Atlas input, legacy catalog input, or both
- dashboard and validation behavior reflect the new surface/scenario model honestly
